# Message Manager: Structure and Token Counting

## 1. The Fundamental Insight: Messages = Memory

The LLM is stateless between calls. Every call is independent — the LLM has no memory of previous interactions. The only way the LLM "remembers" what happened is through the **messages array** passed with each call.

This means:

```
The messages array IS the agent's memory.
If it's not in the messages array, the LLM doesn't know about it.
```

Everything the agent has done, learned, or decided must be represented as messages. Tool results, system instructions, user input, the LLM's own reasoning — all of it lives in this array.

---

## 2. Message Types

The agent uses exactly four message types (following the standard LLM API format):

### 2.1 System Message

```json
{
  "role": "system",
  "content": "You are a helpful AI assistant. You have access to tools..."
}
```

**Purpose**: Defines the agent's identity, behavior rules, tool usage guidelines, and constraints.

**Rules**:
- Always the first message in the array
- Exactly one system message per conversation (some APIs support multiple; MVP uses one)
- Content is the system prompt — the most important design artifact in the entire harness
- Never generated by the LLM; always provided by the harness at initialization

**What goes in it**:
- Agent identity and role
- Behavioral rules (be concise, use tools, verify results)
- Tool usage guidelines (when to use which tool)
- Output format requirements
- Safety constraints
- Error handling guidance

### 2.2 User Message

```json
{
  "role": "user",
  "content": "Find all TODO comments in the codebase and create a report."
}
```

**Purpose**: The task or question the user wants the agent to address.

**Rules**:
- Always the second message (after system)
- Contains the initial task description
- Can also be injected mid-conversation for human-in-the-loop interactions (future)

### 2.3 Assistant Message

```json
{
  "role": "assistant",
  "content": "Let me search the codebase for TODO comments.",
  "tool_calls": [
    {
      "id": "toolu_001",
      "type": "function",
      "function": {
        "name": "search_code",
        "arguments": "{\"pattern\": \"TODO\", \"file_pattern\": \"*.py\"}"
      }
    }
  ]
}
```

**Purpose**: The LLM's response — either a final text answer, or a request to use tools (with optional explanatory text).

**Rules**:
- Generated by the LLM in each loop iteration
- May contain text, tool calls, or both
- `tool_calls` is an array — the LLM can request multiple tools in one response
- The text in `content` is the LLM's reasoning; it's shown to the user but also serves as the LLM's own "thinking out loud"

**Content field nuances**:
- When the LLM calls tools, `content` may be `null` or the LLM's reasoning text
- Some LLMs always include text; some omit it when calling tools
- The harness should handle both cases

### 2.4 Tool Result Message

```json
{
  "role": "tool",
  "tool_call_id": "toolu_001",
  "content": "Found 15 TODO comments in 8 files:\n  - src/main.py:42: TODO: Add error handling\n  ..."
}
```

**Purpose**: The result of executing a tool requested by the LLM.

**Rules**:
- Always follows an assistant message that contained the corresponding `tool_call`
- The `tool_call_id` must match exactly — this is how the LLM correlates results with requests
- Multiple tool results can follow a single assistant message (one per tool call)
- Tool results should be ordered to match the tool_calls order

## 3. Message Array Structure

The messages array has a strict structure:

```
Index 0:  system      (harness-provided, once at initialization)
Index 1:  user        (the initial task)
--- loop iterations ---
Index 2:  assistant   (LLM response: text + optional tool_calls)
Index 3:  tool        (tool result for toolu_001)
Index 4:  tool        (tool result for toolu_002)  -- if multiple tools
Index 5:  assistant   (LLM response after seeing tool results)
Index 6:  tool        ...
Index 7:  assistant   (final answer — text only, no tool_calls)
```

### 3.1 Invariants

The message array must always satisfy:

1. **System first**: A system message is always at index 0
2. **User second**: A user message is always at index 1
3. **Alternation**: After the user message, roles alternate: assistant → tool* → assistant → tool* → ...
4. **Tool follows assistant**: Every tool message must be preceded by an assistant message containing a matching tool_call
5. **Final message is assistant**: The conversation ends with an assistant message (the final answer)
6. **No consecutive assistants**: Cannot have two assistant messages without tool results between them

### 3.2 Example: A Complete Conversation

```json
[
  {
    "role": "system",
    "content": "You are a coding assistant. Use tools to read, write, and search files."
  },
  {
    "role": "user",
    "content": "Create a hello.py file that prints 'Hello, world!'"
  },
  {
    "role": "assistant",
    "content": "I'll create that file for you.",
    "tool_calls": [
      {"id": "toolu_001", "type": "function", "function": {"name": "write_file", "arguments": "{\"path\": \"hello.py\", \"content\": \"print('Hello, world!')\\n\"}"}}
    ]
  },
  {
    "role": "tool",
    "tool_call_id": "toolu_001",
    "content": "Success. Wrote 22 bytes to hello.py."
  },
  {
    "role": "assistant",
    "content": "Let me verify the file was created correctly.",
    "tool_calls": [
      {"id": "toolu_002", "type": "function", "function": {"name": "read_file", "arguments": "{\"path\": \"hello.py\"}"}}
    ]
  },
  {
    "role": "tool",
    "tool_call_id": "toolu_002",
    "content": "print('Hello, world!')\n"
  },
  {
    "role": "assistant",
    "content": "Done! I've created hello.py with the expected content. The file contains exactly one line that prints 'Hello, world!'."
  }
]
```

## 4. Building the Messages Array

### 4.1 The MessageManager Class

```python
class MessageManager:
    def __init__(self, system_prompt: str):
        self.messages: list[dict] = []
        self.add_system(system_prompt)

    def add_system(self, content: str) -> None:
        """Add or replace the system message. Should be first."""
        if self.messages and self.messages[0]["role"] == "system":
            self.messages[0] = {"role": "system", "content": content}
        else:
            self.messages.insert(0, {"role": "system", "content": content})

    def add_user(self, content: str) -> None:
        """Add a user message."""
        self.messages.append({"role": "user", "content": content})

    def add_assistant(self, response: LLMResponse) -> None:
        """Add the LLM's response, including any tool calls."""
        msg = {"role": "assistant", "content": response.text}
        if response.tool_calls:
            msg["tool_calls"] = [
                {
                    "id": tc.id,
                    "type": "function",
                    "function": {
                        "name": tc.name,
                        "arguments": json.dumps(tc.arguments)
                    }
                }
                for tc in response.tool_calls
            ]
        self.messages.append(msg)

    def add_tool_result(self, result: ToolResult) -> None:
        """Add a tool execution result."""
        self.messages.append({
            "role": "tool",
            "tool_call_id": result.tool_call_id,
            "content": result.content
        })

    def to_list(self) -> list[dict]:
        """Return the full message list for LLM API calls."""
        return self.messages

    def to_api_format(self) -> list[dict]:
        """Convert to API-compatible format (may differ by provider)."""
        return self.messages
```

### 4.2 Validation

The message manager can validate invariants before LLM calls:

```python
def validate(self) -> list[str]:
    """Check message array invariants. Returns list of violations."""
    errors = []

    if not self.messages:
        errors.append("Empty message array")
        return errors

    if self.messages[0]["role"] != "system":
        errors.append("First message must be system")

    # Check tool_call_ids match
    tool_call_ids = set()
    for i, msg in enumerate(self.messages):
        if msg["role"] == "assistant" and "tool_calls" in msg:
            for tc in msg["tool_calls"]:
                tool_call_ids.add((i, tc["id"]))

        if msg["role"] == "tool":
            tid = msg.get("tool_call_id")
            if not tid:
                errors.append(f"Tool message at index {i} missing tool_call_id")

    return errors
```

## 5. Token Counting

### 5.1 Why Count Tokens?

LLMs have finite context windows:
- Claude: 200K tokens
- GPT-4o: 128K tokens
- GPT-4-turbo: 128K tokens
- Gemini: 1M+ tokens (some models)

The messages array grows with every iteration. Without monitoring, it can silently overflow the context window, causing:
- Truncation (LLM loses early context — the system prompt or user's original request)
- Errors (API rejects the request)
- Degraded quality (LLM forgets important information)

### 5.2 Approximate Counting

For MVP, use character-based approximation:

```python
def estimate_tokens(text: str) -> int:
    """Quick token estimate: ~4 characters per token for English text."""
    return len(text) // 4

def estimate_message_tokens(messages: list[dict]) -> int:
    """Estimate total tokens across all messages."""
    total = 0
    for msg in messages:
        content = msg.get("content") or ""
        total += estimate_tokens(content)

        # Tool calls have overhead
        if "tool_calls" in msg:
            for tc in msg["tool_calls"]:
                total += estimate_tokens(json.dumps(tc))
                total += 50  # Per-tool-call overhead estimate

    return total
```

### 5.3 Exact Counting

For production, use the model's tokenizer:

```python
import tiktoken  # For OpenAI models

def count_tokens_openai(messages: list[dict], model: str) -> int:
    encoding = tiktoken.encoding_for_model(model)
    total = 0
    for msg in messages:
        total += 4  # Message overhead
        for key, value in msg.items():
            if isinstance(value, str):
                total += len(encoding.encode(value))
            elif isinstance(value, list):
                total += len(encoding.encode(json.dumps(value)))
    total += 2  # Reply priming
    return total
```

For Anthropic models, use the `anthropic` package's token counter (when available) or approximate.

### 5.4 Token Budget Management

```python
class TokenBudget:
    def __init__(self, max_tokens: int, warning_threshold: float = 0.8):
        self.max_tokens = max_tokens
        self.warning_threshold = warning_threshold
        self.warning_issued = False

    def check(self, messages: list[dict]) -> TokenStatus:
        used = estimate_message_tokens(messages)
        remaining = self.max_tokens - used

        if remaining < 0:
            return TokenStatus.OVER_CAPACITY

        if not self.warning_issued and used > self.max_tokens * self.warning_threshold:
            self.warning_issued = True
            return TokenStatus.NEAR_CAPACITY

        return TokenStatus.OK


class TokenStatus(Enum):
    OK = "ok"
    NEAR_CAPACITY = "near_capacity"
    OVER_CAPACITY = "over_capacity"
```

## 6. Message Growth Patterns

Understanding how messages grow helps predict when context will overflow:

### 6.1 Growth per Iteration

| Component | Typical Size (tokens) |
|---|---|
| System prompt | 500 - 5,000 |
| User message | 50 - 500 |
| Assistant reasoning (per iter) | 100 - 1,000 |
| Tool definitions (sent each call) | 50 - 200 per tool |
| Tool call (per tool) | 50 - 200 |
| Tool result (per tool) | 100 - 50,000+ |

### 6.2 Example Budget

```
System prompt:      2,000 tokens
User message:         200 tokens
Iteration 1:
  Assistant:          300 tokens
  Tool call:          100 tokens
  Tool result:       1,000 tokens
Iteration 2:
  Assistant:          400 tokens
  Tool call × 3:      300 tokens
  Tool result × 3:   3,000 tokens
...
Final answer:         500 tokens
─────────────────────────────
Total:              ~7,800 tokens for a simple 2-iteration task
```

A complex task with 15 iterations and large tool results can hit 100K+ tokens easily.

### 6.3 The Context Ceiling

For MVP (200K context window), overflow is unlikely for typical tasks. But it's the first problem you'll hit as tasks get more complex or tools return larger results. Context management (document 02) addresses this.

## 7. The System Prompt Design Connection

The Message Manager stores the system prompt, but doesn't define it. The prompt is designed elsewhere (in Config). However, the Message Manager must handle:

- **Prompt updates**: If the system prompt changes mid-conversation (e.g., after compaction)
- **Prompt injection**: Adding temporary system instructions (e.g., "You have 3 iterations remaining")
- **Multi-part system messages**: Some providers support multiple system blocks; MVP uses one

## 8. MVP Implementation

```python
class MessageManager:
    """Manages the message array — the agent's working memory."""

    def __init__(self, system_prompt: str):
        self.messages: list[dict] = []
        self.token_budget = TokenBudget(max_tokens=200_000)

        # Initialize with system message
        self.messages.append({
            "role": "system",
            "content": system_prompt
        })

    def add_user(self, content: str):
        self.messages.append({"role": "user", "content": content})

    def add_assistant(self, text: str | None, tool_calls: list | None = None):
        msg: dict = {"role": "assistant"}
        if text:
            msg["content"] = text
        else:
            msg["content"] = None

        if tool_calls:
            msg["tool_calls"] = [
                {
                    "id": tc.id,
                    "type": "function",
                    "function": {
                        "name": tc.name,
                        "arguments": json.dumps(tc.arguments)
                    }
                }
                for tc in tool_calls
            ]

        self.messages.append(msg)

    def add_tool_result(self, tool_call_id: str, content: str):
        self.messages.append({
            "role": "tool",
            "tool_call_id": tool_call_id,
            "content": content
        })

    def add_system_notice(self, notice: str):
        """Inject a system notice as a user message with [System] prefix.
           The LLM treats these as important meta-instructions."""
        self.messages.append({
            "role": "user",
            "content": f"[System Notice] {notice}"
        })

    def estimate_tokens(self) -> int:
        """Estimate total tokens in the message array."""
        total = 0
        for msg in self.messages:
            total += len(json.dumps(msg)) // 4  # Rough estimate
        return total

    def check_context(self) -> TokenStatus:
        """Check if we're approaching the context limit."""
        return self.token_budget.check(self.messages)

    def to_list(self) -> list[dict]:
        return self.messages

    def last_n_messages(self, n: int) -> list[dict]:
        """Get the last N messages (preserving system message)."""
        if n >= len(self.messages) - 1:
            return self.messages
        return [self.messages[0]] + self.messages[-(n):]
```

## 9. MVP Decisions

| Decision | Rationale |
|---|---|
| Four message types only | Matches all major LLM APIs exactly |
| Single system message | Simpler than multi-block; sufficient for MVP |
| Character-based token estimate | Good enough for warnings; exact counting needs model-specific tokenizers |
| No message deletion | Messages are append-only for MVP; compaction comes later |
| System notices as user messages | Simple injection mechanism; LLM treats [System] prefix seriously |
| JSON-formatted tool calls | Matches Anthropic/OpenAI API formats exactly |
| `tool_call_id` matching enforced | Critical for LLM to correlate results with requests |
