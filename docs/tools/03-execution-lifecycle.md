# Tool Execution Lifecycle

## 1. Overview: The Five-Stage Pipeline

When the LLM decides to call a tool, the harness executes this pipeline:

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   RECEIVE    │───▶│   VALIDATE   │───▶│   EXECUTE    │───▶│    FORMAT    │───▶│    RETURN    │
│              │    │              │    │              │    │              │    │              │
│ Parse LLM's  │    │ Check name,  │    │ Call handler │    │ Structure    │    │ Inject into  │
│ tool_call    │    │ schema,      │    │ function     │    │ result for   │    │ messages     │
│ request      │    │ permissions  │    │ with args    │    │ LLM context  │    │ array        │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

Each stage has exactly one responsibility. Failures cascade forward (an error at any stage becomes a formatted error result).

---

## 2. Stage 1: Receive

### 2.1 What Happens

The LLM returns a response containing tool calls. The harness extracts them:

```python
# LLM response structure
response = {
    "id": "msg_abc123",
    "role": "assistant",
    "content": [  # may contain text AND tool_use blocks
        {"type": "text", "text": "Let me read that file for you."},
        {"type": "tool_use", "id": "toolu_001", "name": "read_file",
         "input": {"path": "/tmp/data.json"}}
    ]
}
```

### 2.2 Key Details

- **Multiple tool calls per response**: The LLM may request several tools at once. Execute them sequentially (or in parallel — see architecture decisions).
- **Text + tool calls**: The LLM often explains its reasoning (text) AND requests tools in the same response. Both are preserved in the messages array.
- **Tool call IDs**: Each tool call has a unique ID (generated by the LLM). This ID must appear in the corresponding tool result message — it's how the LLM correlates requests with results.
- **Order matters**: If the LLM calls tools A, B, C, execute them in that order. Results are appended in the same order. The LLM reads them in order.

### 2.3 Data Structure

```typescript
interface ToolCall {
  id: string              // unique ID from LLM (e.g., "toolu_001")
  name: string            // tool name the LLM wants to invoke
  arguments: Record<string, unknown>  // the arguments as a parsed object
}
```

## 3. Stage 2: Validate

### 3.1 What Happens

Before executing anything, the harness validates the request:

```
1. Tool exists?     → No  → "Tool 'X' not found"
2. Arguments valid?  → No  → "Invalid arguments: ..."
3. Permissions OK?   → No  → "Permission denied: ..."
```

### 3.2 Tool Lookup

```python
def validate_tool_exists(name: str, registry: dict) -> Tool:
    if name not in registry:
        # Generate helpful error: suggest similar names
        suggestions = find_similar_names(name, registry.keys())
        msg = f"Tool '{name}' not found."
        if suggestions:
            msg += f" Did you mean: {', '.join(suggestions)}?"
        raise ToolNotFoundError(msg)
    return registry[name]
```

### 3.3 Schema Validation

```python
def validate_arguments(schema: dict, args: dict) -> dict:
    validator = JSONSchemaValidator(schema)
    errors = validator.validate(args)
    if errors:
        # Format errors clearly for the LLM
        msg = "Invalid arguments:\n"
        for err in errors:
            msg += f"  - {err.path}: {err.message}\n"
        msg += f"\nExpected schema: {json.dumps(schema, indent=2)}"
        raise SchemaValidationError(msg)
    return args  # valid + any defaults applied
```

Schema validation should:
1. Check that all `required` fields are present
2. Verify types match (string, number, boolean, array, object)
3. Apply `default` values for missing optional fields
4. Reject unknown fields (or silently ignore — design decision)
5. Check enum values, min/max constraints

### 3.4 Permission Check

```python
def check_permissions(tool: Tool, args: dict) -> bool:
    if tool.requires_approval:
        # For MVP: auto-deny destructive operations, auto-approve reads
        # For production: prompt user or check policy
        pass
    return True
```

### 3.5 Validation Failure = Tool Result

When validation fails, don't crash the agent loop. Instead, return the error as a formatted tool result. The LLM reads the error and self-corrects:

```
LLM:  calls "read_fil" (typo)
Harness: returns error → "Tool 'read_fil' not found. Did you mean: read_file?"
LLM:  reads error, retries with correct name "read_file"
```

## 4. Stage 3: Execute

### 4.1 What Happens

Call the registered handler function with validated arguments:

```python
def execute_tool(tool: Tool, args: dict) -> Any:
    try:
        result = tool.handler(**args)
        # Handler might be sync or async
        if is_coroutine(result):
            result = await result
        return result
    except Exception as e:
        # Any exception becomes a ToolExecutionError
        raise ToolExecutionError(tool.name, args, e)
```

### 4.2 Execution Concerns

**Timeout**: Every tool call should have a timeout. Infinite loops or hung network calls kill the agent.

```python
result = await asyncio.wait_for(
    tool.handler(**args),
    timeout=tool.timeout or DEFAULT_TIMEOUT  # e.g., 30 seconds
)
```

**Cancellation**: The agent loop should be able to cancel in-flight tool executions (user interrupts, error in parallel tool, etc.).

**Isolation**: The handler's exceptions should not crash the harness. Catch everything and convert to `ToolExecutionError`.

### 4.3 The Handler Signature

```typescript
type ToolHandler = (args: Record<string, unknown>) => unknown | Promise<unknown>
```

The handler does the actual work. It can:
- Read/write files
- Make HTTP requests
- Run shell commands
- Query databases
- Call other APIs

The handler is the only part of the tool system that has side effects. Everything else (validation, formatting) is pure bookkeeping.

---

## 5. Stage 4: Format

### 5.1 What Happens

Convert the raw execution result (could be anything — string, dict, bytes, None) into a string suitable for the LLM's context:

```python
def format_result(raw: Any) -> str:
    if raw is None:
        return "Success (no output)."

    if isinstance(raw, str):
        return raw

    if isinstance(raw, (dict, list)):
        return json.dumps(raw, indent=2, ensure_ascii=False)

    return str(raw)
```

### 5.2 Formatting Rules

1. **Never return empty string**: If the tool succeeded but produced no output, return `"Success (no output)."` — the LLM needs positive confirmation.
2. **Truncate large results**: If output exceeds N characters (e.g., 50K), truncate and append `"... [truncated at N characters]"`.
3. **Include structure hints**: For JSON results, pretty-print. For binary results, show size/type instead of raw bytes.
4. **Add metadata**: Consider prefixing with tool name: `"[read_file] Returned 150 lines."` — helps the LLM track which result is which when multiple tools are called.

### 5.3 The ToolResult Message

The formatted result becomes a message in the standard tool result format:

```json
{
  "role": "tool",
  "tool_call_id": "toolu_001",
  "content": "Success. File contents:\n{ ... truncated ... }"
}
```

The `tool_call_id` must match the ID from the LLM's tool call. This is how the LLM knows which request this result corresponds to.

---

## 6. Stage 5: Return (Context Injection)

### 6.1 What Happens

The formatted tool result message is appended to the agent's message array. On the next loop iteration, the LLM sees:

```
messages = [
    {role: "system", content: "..."},
    {role: "user", content: "Read /tmp/data.json"},
    {role: "assistant", content: "Let me read that file.", tool_calls: [...]},
    {role: "tool", tool_call_id: "toolu_001", content: "Success. File contents: ..."},
    // ← Next LLM call happens here. LLM sees all of the above.
]
```

The agent loop continues. The LLM now has the tool result in context and can decide the next action.

### 6.2 Multiple Tool Calls

If the LLM called tools A, B, C in one response:

```
messages = [
    ...,
    {role: "assistant", tool_calls: [
        {id: "tc_a", name: "X"}, {id: "tc_b", name: "Y"}, {id: "tc_c", name: "Z"}
    ]},
    {role: "tool", tool_call_id: "tc_a", content: "result A"},
    {role: "tool", tool_call_id: "tc_b", content: "result B"},
    {role: "tool", tool_call_id: "tc_c", content: "result C"},
    // ← Next LLM call. LLM sees all three results.
]
```

All tool results from a single response are available to the LLM in the next call. This enables the LLM to call several independent tools (e.g., read three files) and then synthesize the results.

## 7. The Complete Pipeline (Pseudocode)

```python
async def execute_tool_call(tool_call: ToolCall, registry: ToolRegistry) -> ToolResult:
    # Stage 1: Receive (already parsed from LLM response)
    tool_name = tool_call.name
    tool_args = tool_call.arguments
    tool_id = tool_call.id

    # Stage 2: Validate
    tool = registry.get(tool_name)
    if tool is None:
        return ToolResult.error(tool_id, f"Tool '{tool_name}' not found")

    valid, errors = tool.validate_schema(tool_args)
    if not valid:
        return ToolResult.error(tool_id, f"Invalid arguments: {errors}")

    if not registry.check_permissions(tool, tool_args):
        return ToolResult.error(tool_id, f"Permission denied: '{tool_name}'")

    # Stage 3: Execute
    try:
        raw_result = await asyncio.wait_for(
            tool.handler(**tool_args),
            timeout=tool.timeout
        )
    except asyncio.TimeoutError:
        return ToolResult.error(tool_id, f"Tool '{tool_name}' timed out after {tool.timeout}s")
    except Exception as e:
        return ToolResult.error(tool_id, f"Tool '{tool_name}' failed: {e}")

    # Stage 4: Format
    formatted = format_result(raw_result)

    # Stage 5: Return (caller injects into messages)
    return ToolResult.success(tool_id, formatted)
```

## 8. Edge Cases the MVP Must Handle

| Edge Case | Behavior |
|---|---|
| LLM calls unknown tool | Return error with name suggestions |
| LLM passes wrong argument types | Return schema validation error with expected schema |
| LLM passes extra unknown args | Silently ignore them (or reject — design decision) |
| Tool handler raises exception | Catch, format as error result, feed back to LLM |
| Tool handler hangs | Timeout → return timeout error |
| Tool returns very large output | Truncate at limit, append truncation notice |
| Tool returns None | Return "Success (no output)" |
| Tool returns binary data | Return size + type description, not raw bytes |
| LLM calls same tool repeatedly | Loop detection — consider tracking call count per tool |
| Multiple tools in one response | Execute sequentially, append all results before next LLM call |

## 9. MVP Implementation Sketch

```python
class ToolExecutor:
    def __init__(self, registry, timeout=30):
        self.registry = registry
        self.timeout = timeout

    async def execute(self, tool_call: ToolCall) -> ToolResultMessage:
        # 1. Lookup
        tool = self.registry.get(tool_call.name)

        # 2. Validate
        if tool is None:
            return ToolResultMessage(
                tool_call_id=tool_call.id,
                content=f"Tool '{tool_call.name}' not found. "
                        f"Available: {', '.join(self.registry.names())}"
            )

        try:
            tool.validate(tool_call.arguments)
        except ValidationError as e:
            return ToolResultMessage(
                tool_call_id=tool_call.id,
                content=str(e)
            )

        # 3. Execute
        try:
            result = await asyncio.wait_for(
                tool.handler(**tool_call.arguments),
                timeout=tool.timeout or self.timeout
            )
        except asyncio.TimeoutError:
            return ToolResultMessage(
                tool_call_id=tool_call.id,
                content=f"Timeout: '{tool_call.name}' exceeded {tool.timeout}s limit"
            )
        except Exception as e:
            return ToolResultMessage(
                tool_call_id=tool_call.id,
                content=f"Error in '{tool_call.name}': {e}"
            )

        # 4. Format
        return ToolResultMessage(
            tool_call_id=tool_call.id,
            content=format_output(result)
        )


def format_output(result: Any) -> str:
    if result is None:
        return "Success."
    if isinstance(result, str):
        return truncate(result, max_chars=50_000)
    if isinstance(result, (dict, list)):
        return truncate(json.dumps(result, indent=2), max_chars=50_000)
    return truncate(str(result), max_chars=50_000)


def truncate(text: str, max_chars: int) -> str:
    if len(text) <= max_chars:
        return text
    return text[:max_chars] + f"\n... [truncated at {max_chars} characters]"
```
